## 3. Pages authentifiées — plan de wiring

Les 3 URLs auth (Léa /accueil, Gabrielle parcours, /mes-documents) ne sont pas dans le scope de ce baseline car elles nécessitent une session valide.

Plan retenu (post-merge) :

1. Ajouter à `scripts/lighthouse/` un script `login.js` (Puppeteer) qui :
   - Charge `/login`, remplit les champs, soumet, attend la redirection.
   - Lit les creds depuis `NAMI_TEST_PATIENT_EMAIL` / `NAMI_TEST_PATIENT_PASSWORD` (patient) et `NAMI_TEST_CARE_EMAIL` / `NAMI_TEST_CARE_PASSWORD` (soignant).
2. Référencer `puppeteerScript` dans `.lighthouserc.json` sous `collect.settings`.
3. Câbler dans le workflow GitHub via secrets repo `LIGHTHOUSE_PATIENT_*` / `LIGHTHOUSE_CARE_*`.
4. Le wrapper `run-audit.sh --with-auth` détecte déjà la présence des env vars et lance les runs authentifiés.

Pourquoi pas dans cette PR : la base auth doit être seedée stable en prod (Léa + Gabrielle existent — cf. `MEMORY.md` `reference_demo_creds_lea`), mais commiter des creds — même de démo — dans un workflow CI sans secrets configurés en amont est une mauvaise hygiène. Suivi : `F-PERF-LIGHTHOUSE-AUTH-FLOW`.

## 4. Violations identifiées + quick wins

### 4.a — `/` (landing patient) · ❌ NO_LCP

Lighthouse `largest-contentful-paint`.errorMessage = `NO_LCP`.

| Élément | Constat |
|---------|---------|
| Top byte request | `/nami-mascot.png` — **283 KiB**, sans `priority` ni `sizes` détectés |
| Total page | 911 KiB / 62 requests / 345 nodes DOM |
| Critical request chains | 2 |
| Largest contentful element | non identifié par LH |

**Quick win #1 (P0 launch)** — Marquer le hero comme LCP candidate côté SSR :
- Hero `<h1>` rendu côté serveur (déjà le cas), dimensions stables, sans transform/opacity initial.
- `nami-mascot.png` → `next/image` avec `priority`, `sizes` explicites (`(max-width: 768px) 60vw, 400px`) et conversion AVIF/WebP via Next image optimizer (potentiel ~200 KB gain). 
- Vérifier qu'aucun `opacity:0` initial (ScrollReveal) n'est appliqué sur l'élément qui devrait être LCP avant peinture.

### 4.b — `/login` · ⚠ SEO 66 (intentionnel)

`is-crawlable: 0` — la page expose `<meta name="robots" content="noindex">`. **Comportement attendu** : on ne veut pas indexer une page de login. Pas un fix à faire.

**Action** : ignorer cet audit pour `/login` uniquement (`assertMatrix` côté `.lighthouserc`) plutôt que skipper l'audit globalement.

### 4.c — `app.namipourlavie.com/` · ⚠ A11y 93

| Audit | Score | Constat |
|-------|-------|---------|
| `color-contrast` | 0 | Texte muted `#6B7280` sur fond `#FAFAF8` → ratio 4.6:1 (limite WCAG AA pour <18 pt). Probable sur un élément small/disclaimer. |
| `image-redundant-alt` | 0 | Une image a un `alt` qui répète un texte adjacent (anti-pattern WCAG). |
| `skip-link` | 0 | Le skip link n'est pas focusable au clavier (probablement `display:none` au lieu de `clip-path`). |

**Quick wins #2–4 (P1 launch)** :
- Hausser le texte muted à `#4B5563` sur fond crème (5.7:1).
- Auditer les `<img alt>` du hero pro et retirer ceux qui dupliquent le titre adjacent.
- Skip link : `position: absolute; left: -9999px;` + `:focus { left: 0; }` au lieu de `display:none`.

### 4.d — `app.namipourlavie.com/` · ⚠ Opportunités perf (non-bloquantes)

| Opportunité | Économie estimée |
|-------------|------------------|
| `modern-image-formats` (AVIF/WebP) | **~265 KiB** |
| `uses-responsive-images` | ~281 KiB |
| `unused-javascript` | ~85 KiB / 80 ms |
| `legacy-javascript` (polyfills inutiles modernes) | ~31 KiB |
| `uses-long-cache-ttl` | 2 ressources sans cache long |

**Quick win #5 (P1 launch)** — Activer Next Image partout sur la landing pro (extension de #1) — un seul fix résout `modern-image-formats` + `uses-responsive-images` + une partie d'`unused-javascript`.
