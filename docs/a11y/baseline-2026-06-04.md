## Détail des violations

### `color-contrast` (WCAG 2.1 AA — 1.4.3)

Unique règle violée, partout. Le critère AA exige un ratio de **4.5:1** pour le
texte standard et **3:1** pour le texte large. Le design system Nami utilise
beaucoup de gris désaturé `#6B7280` sur `#FAFAF8` (ratio 4.6:1, à la limite),
de l'indigo clair `#5B4EC4`/`#A78BFA` sur fond crème, et des tags pastel sur
bord blanc — combinaisons qui flirtent avec le seuil sans toujours le passer.

- Doc règle axe : https://dequeuniversity.com/rules/axe/4.11/color-contrast
- Tags : `wcag2aa`, `wcag143`, `EN-301-549`, `RGAAv4 3.2.1`

## P1 — Plan de fix post-launch (si bandwidth)

Aucune violation critical → la conformité **WCAG 2.1 A** est atteinte sur les
6 pages auditées. Le gap réel est sur **WCAG 2.1 AA — contraste**, qui peut
être lissé en 1-2 sprints sans toucher la palette d'identité.

1. **P1.1 — Auditer les tokens texte muted (`text-muted`, `text-subtle`)**
   Probablement la source principale. Tester en relevant à `#5B6478` (ratio ≥ 5:1
   sur `#FAFAF8`). Impact attendu : −30 à −40 nœuds sur les 53.
2. **P1.2 — Boutons secondaires & badges pastel sur les deux landings**
   Cibler en priorité les badges section (catégorie, statut) et les CTAs
   secondaires qui utilisent des fonds très clairs.
3. **P1.3 — Liens muted dans les listes (`/parcours/[id]` + drawer)**
   Petits libellés (métadonnées, dates, sources) — souvent en `#9CA3AF`. Passer
   à `#6B7280` minimum.
4. **P1.4 — Activer le scan a11y en CI sur PR**
   Ajouter le project `a11y` au workflow `e2e.yml` (déjà en place pour la suite
   chromium). Bloquer sur **critical seulement** au début, puis remonter le
   seuil à serious une fois P1.1-P1.3 lissés.
5. **P1.5 — Compléter le scan**
   - Cockpit `/evenements` une fois les creds soignant figés (variable env
     `A11Y_SOIGNANT_*` documentée plus haut).
   - Pages légales `/cgu`, `/confidentialite`, `/mentions-legales` (requis
     pre-launch C9, à ajouter en spec dédiée).

## Limitations connues

- **Scope** : audit statique (axe-core ne couvre pas le clavier, le screen reader
  ni les interactions complexes). Le baseline ne remplace pas un audit RGAA
  manuel par un expert.
- **Cockpit `/evenements` non scanné** : les creds soignant documentés dans
  `CLAUDE.md` peuvent être stales en prod. Spec en `test.skip` graceful jusqu'à
  fourniture d'un compte démo soignant stable.
- **Rate-limit prod sur `/login`** : ré-exécuter le baseline complet en boucle
  serrée déclenche un throttle (toast d'erreur, navigation bloquée). Le pattern
  storageState (1 login → réutilisé par 5 specs) atténue, mais espacer les runs
  de ~2 min si nécessaire.
- **SSR race Zustand persist patient** : `src/app/(patient)/layout.tsx` redirige
  parfois vers `/login` sur cold-load (cf mémoire `feedback_zustand_ssr_race`).
  Les specs intègrent un retry × 3 — si toutes échouent, fixer `hasHydrated`
  côté layout (toujours en dette).
